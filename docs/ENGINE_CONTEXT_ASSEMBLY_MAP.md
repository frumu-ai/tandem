# Engine Context Assembly Map

TAN-186 inventory. Last traced from `main` at `3094ac42` on 2026-06-09.

This document maps the places Tandem assembles provider-facing model context. It is a behavior-free reference for follow-up context hygiene work.

## Final Provider Boundary

All ordinary chat, workflow, automation, routine, and coder-worker paths converge on `EngineLoop::run_prompt_async_with_context`, then on the provider registry stream call.

- Owner: `crates/tandem-core/src/engine_loop/prompt_execution.rs:240`
- Final send: `crates/tandem-core/src/engine_loop/prompt_execution.rs:660`
- Provider interface: `crates/tandem-providers/src/lib_parts/part01.rs:659`

Assembly order in the core loop:

1. Select `ChatHistoryProfile` from requested context mode.
2. Load session history with `load_chat_history`.
3. Attach runtime image attachments to the latest user message on the first iteration.
4. Insert the Tandem runtime system prompt and active agent system prompt.
5. Append follow-up context generated by tool-loop guards or repair paths.
6. Run the server prompt context hook, which may append identity, memory scope, KB grounding, docs, or global memory blocks.
7. Select tools and schemas.
8. Stream `Vec<ChatMessage>` plus tool mode and schemas through `ProviderRegistry::stream_for_provider`.

Provider wire shape is adapter-specific:

- OpenAI Chat Completions normalizes system messages and sends `messages`, optional `tools`, and `stream_options.include_usage`; see `crates/tandem-providers/src/lib_parts/part02.rs:113`.
- OpenAI Responses splits system content into `instructions` and sends non-system messages as `input`; see `crates/tandem-providers/src/lib_parts/part02.rs:557` and `crates/tandem-providers/src/lib_parts/part03.rs:49`.
- Anthropic currently maps `ChatMessage` to role/content message objects; see `crates/tandem-providers/src/lib_parts/part02.rs:1293`.

## Core History And Tool Results

Owner: `crates/tandem-core/src/engine_loop/prompt_runtime.rs:151`

Injected sources:

- Persisted session messages from `Storage::get_session`.
- Text and reasoning parts are flattened into provider message content.
- Tool invocation parts are converted to a textual summary by `summarize_tool_invocation_for_history`.
- `mcp_list` results are compacted to connected server names, counts, and a sample of up to 40 tools.
- Image runtime attachments are attached separately as `ChatAttachment::ImageUrl`.

Timing:

- Rebuilt on every provider iteration, before the system prompt is inserted.
- Runtime image attachments are only attached on iteration 1.

Raw artifact preservation:

- Raw session messages and tool invocation payloads remain in storage.
- Provider history receives the flattened or summarized projection only.

Token estimation:

- History profile selection emits `historyCharCount`.
- Standard and compact modes use character budgets, not tokenizer-aware budgets.
- Full context mode disables history compaction, but later prompt augmentation and provider wrappers still apply.

## Session Chat And KB Grounding

Owner: `crates/tandem-server/src/http/sessions.rs:1051`

Injected sources:

- User request parts from `SendMessageRequest`.
- Optional strict KB grounding policy derived by the session handler.
- If a strict text-only KB request can be answered directly, the handler calls the KB MCP `answer_question` path and persists the result without entering the core provider loop.
- Otherwise, the session handler sets a KB grounding policy, forces required tools, and lets the prompt context hook inject a KB grounding policy block.

Timing:

- KB policy is derived before `run_prompt_async_with_context`.
- Direct KB answer happens before provider context assembly.
- Policy-based KB grounding is injected during prompt hook augmentation.

Raw artifact preservation:

- Direct KB answer messages are persisted as session messages.
- Policy-based KB tool use is preserved through normal tool invocation/result message parts.

Token estimation:

- Direct KB answer path does not use the provider prompt estimator.
- Policy-based KB block contributes to the post-hook provider message char count estimate.

## Prompt Context Hook

Owner: `crates/tandem-server/src/app/state/mod.rs:1555`

Registered by `AppState` at `crates/tandem-server/src/app/state/app_state_impl_parts/part01.rs:511`.

Injected sources:

- Identity block from effective config and active agent profile.
- Memory scope block for session, project, and workspace.
- Required KB grounding block when the session has a KB grounding policy.
- Embedded guide/docs search hits from `search_embedded_docs`.
- Global memory hits from `MemoryManager::search_global_memory`, filtered by governance visibility.

Timing:

- Runs after the core system prompt and any follow-up context are in the message vector.
- Appends extra system messages.
- Fails open if runtime state is not ready, the run registry has no active run, the query is empty, or memory injection is skipped.

Raw artifact preservation:

- Docs and memory hits stay in their source stores; the hook injects rendered summaries only.
- Global memory rendering is capped by `MEMORY_CONTEXT_CHAR_BUDGET` in `crates/tandem-server/src/app/state/prompt_memory_context.rs`.

Token estimation:

- Hook emits approximate token sizes with `split_whitespace().count()`.
- Final provider usage falls back to char-count estimates if the provider does not report usage.

## Workflow Execution

Owner: `crates/tandem-server/src/workflows.rs:1135`

Injected sources:

- Agent action prompt from `action_spec.with.prompt`, or a generated fallback objective.
- Workflow correlation ID in the form `workflow:{workflow_id}`.
- The action creates a new session and passes a single text user message into the core engine loop.

Timing:

- Workflow prompt construction happens before the engine loop.
- History, system prompt, memory/docs/KB, tool results, and provider wrapper behavior are the same as ordinary sessions once inside the loop.

Raw artifact preservation:

- Workflow run/action state is stored in the workflow layer.
- Provider-facing context only gets the selected action prompt plus normal session projections.

Token estimation:

- No workflow-specific token estimator was found.
- Uses core provider usage or char-count fallback.

## Automation V2

Owners:

- Prompt runtime values and knowledge preflight: `crates/tandem-server/src/app/state/automation/logic_parts/part01.rs:361`
- Node execution prompt assembly: `crates/tandem-server/src/app/state/automation/logic_parts/part05.rs:2067`
- Prompt renderer: `crates/tandem-server/src/app/state/automation/prompting_impl.rs:536`

Injected sources:

- Automation/run/node identity, objective, workspace root, attempt, execution mode, selected tools, required artifacts, write policies, and output contracts.
- Runtime placeholders such as current date and timestamp.
- Upstream node outputs, with optional summary-only degradation.
- Runtime context partitions from automation upstream materialization.
- Knowledge preflight results, workflow learning context, and previous standup context when applicable.
- MCP contract guidance and connector/tool availability guidance.
- Repair briefs and validator-specific final response contracts.

Timing:

- Automation builds a full text prompt before calling `run_prompt_async_with_context`.
- It sets allowed tools, write policy, and auto-approval policy on the session before the prompt call.
- Prompt preflight may rebuild the prompt with summary-only upstream inputs before sending.

Raw artifact preservation:

- Run checkpoints, node outputs, context partitions, artifacts, and workspace writes remain in automation storage or filesystem locations.
- Provider-facing context receives rendered prompt text and compacted upstream summaries.
- Context-write IDs are stripped from injected upstream prompt payloads where the renderer compacts outputs.

Token estimation:

- `build_automation_prompt_preflight` records rendered prompt chars, estimated prompt tokens, and offered tool schema chars before core assembly.
- The estimate is `chars.div_ceil(4)`, with warning/high thresholds at 2,400 and 3,200 estimated tokens.
- This preflight covers the automation-rendered text and offered schemas, but not the later system prompt, chat history, prompt hook additions, provider-specific wire transforms, or image attachments.

## Routines

Owner: `crates/tandem-server/src/app/tasks.rs:1162`

Injected sources:

- Routine prompt from `build_routine_prompt`.
- Selected model from the routine spec when configured.
- Correlation ID in the form `routine:{run_id}`.

Timing:

- Routine prompt construction happens before the engine loop.
- Provider context assembly follows the ordinary session path after the routine message is stored.

Raw artifact preservation:

- Routine run state stores token/cost totals and run metadata.
- Provider-facing prompt is a single rendered routine text message plus normal session context.

Token estimation:

- No routine-specific prompt estimator was found.
- Provider usage events are attributed back to routine runs by correlation ID.

## Coder And Autonomous Workers

Owner: `crates/tandem-server/src/http/coder_parts/part06.rs:99`

Injected sources:

- Managed worker workspace root.
- Canonical repo root.
- Worker kind.
- A coder contract requiring repository inspection, scoped instructions, workspace diffs, validation, and handoff sections.
- Worker-specific task prompt, such as issue fix, validation, PR review, triage, or merge recommendation prompts.

Timing:

- The worker contract prompt is built before `run_prompt_async_with_context`.
- Coder workers request `ContextMode::Full`.
- Issue-fix workers may require tool usage and prewrite inspection requirements.

Raw artifact preservation:

- Coder run, session, and workspace state live outside the provider prompt.
- Provider-facing context gets the rendered worker contract and subsequent session history projections.

Token estimation:

- No coder-specific prompt estimator was found.
- Full context mode preserves all chat history before later provider and hook additions.

## Tool Result Injection

Owner: `crates/tandem-core/src/engine_loop/prompt_runtime.rs:170`

Injected sources:

- Tool invocation name, args, result, and error from persisted message parts.
- Current-iteration tool results also feed follow-up iterations through persisted session history.

Timing:

- Tool results enter model context on the next prompt iteration or later run after being persisted as message parts.
- `followup_context` can add extra user context in the same loop before provider send.

Raw artifact preservation:

- Raw tool result payloads remain in session storage.
- Provider history gets a textual summary, with special compaction for `mcp_list`.

Token estimation:

- Tool result summaries are included in core char counts.
- There is no per-tool token accounting at prompt assembly time.

## Token Accounting Summary

Known accounting paths:

- `context.profile.selected` emits history message count and history chars before system and hook additions.
- Core computes `estimated_prompt_chars` from final `messages` before provider send.
- Providers may report `TokenUsage`.
- If providers omit usage, `provider_usage_token_counts` estimates prompt and completion tokens as chars divided by 4.
- Automation preflight estimates the rendered automation prompt before the engine loop and reports offered tool schema chars.
- Provider usage events are attributed to automation, routine, governance, benchmarking, and agent-team ledgers by session/correlation handlers.

Gaps to close in follow-up work:

- No unified tokenizer-aware estimate at the final provider boundary.
- Automation preflight does not include later core system prompt, history, prompt hook additions, provider wire transforms, or attachments.
- History compaction is char-budget based.
- Prompt hook token sizes are word-count approximations.
- Tool schemas are counted by serialized chars in automation preflight, but not as final provider tokens.
- Image attachments are not included in char/token estimates.
- Direct strict-KB answer path bypasses the model context accounting path by design.
